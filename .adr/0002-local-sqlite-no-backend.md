# ADR-0002: 서버 없이 SQLite 온디바이스 저장

- **상태**: Accepted
- **결정일**: 2026-05-18

---

## 맥락

사용자의 기록 데이터를 어디에 저장할지 결정해야 했다.
MVP 단계에서 백엔드 서버 운영 비용과 복잡도를 최소화하고,
사용자 개인정보 보호를 우선순위로 두었다.

## 결정

클라우드 백엔드 없이 **기기 내 SQLite** 만 사용한다.
`sqflite` 패키지로 `records`, `patterns` 두 테이블을 관리한다.

## 근거

- MVP 단계에서 서버 인프라 비용 없음
- 오프라인에서도 완전히 동작
- 민감한 개인 기록 데이터가 외부 서버에 전송되지 않아 개인정보 보호 강화
- 1인 개발 공수 절감 (서버 API, 인증, 동기화 로직 불필요)

## 대안

| 대안 | 제외 이유 |
|------|-----------|
| Firebase Firestore | 실시간 동기화 장점 있으나 MVP에 과도한 복잡도, 비용 발생 |
| Supabase (PostgreSQL) | 백엔드 서버 운영 필요, 개인정보 데이터 외부 전송 |
| Hive (NoSQL) | 스키마 없는 구조로 관계형 쿼리가 불편 |

## 결과

**긍정적:**
- 완전한 오프라인 동작
- 설정 없이 앱 설치만으로 바로 사용 가능
- 사용자 데이터가 기기 밖으로 나가지 않음

**부정적:**
- 기기 분실 시 데이터 복구 불가 (백업 기능 없음)
- 여러 기기 간 데이터 동기화 불가 (v2 과제)
- 클라우드 기반 분석/학습 개선 어려움

## 향후 고려

v2에서 선택적 클라우드 백업(Firebase Storage 또는 iCloud)을 추가할 수 있다.
이 경우 현재 Repository 패턴으로 설계되어 있어 데이터 소스 교체가 비교적 용이하다.
